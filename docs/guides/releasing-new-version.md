# Releasing a New Version

Guide for cutting a new td release. If you are an automated agent, jump to
[Agent Runbook](#agent-runbook) — it is a non-interactive, copy-paste sequence.

## How It Works

Releases are triggered by **pushing a `v*` tag**. The tag push runs
[`.github/workflows/release.yml`](../../.github/workflows/release.yml), which has
two jobs:

1. **`goreleaser`** — runs [GoReleaser](https://goreleaser.com/) (config:
   [`.goreleaser.yml`](../../.goreleaser.yml)). Builds binaries for
   darwin/linux × amd64/arm64, creates the GitHub release with the archives +
   `checksums.txt`, and generates the **GitHub release notes** from commit
   subjects since the last tag (excluding `docs:`/`test:`/`ci:`/`chore:`).
2. **`update-homebrew-tap`** — runs after `goreleaser`. Downloads the source
   tarball for the new tag, computes its sha256, and rewrites the tag URL +
   sha256 in `Formula/td.rb` in the `marcus/homebrew-tap` repo, then commits and
   pushes. The formula builds td **from source** (this avoids macOS Gatekeeper
   warnings, which is why GoReleaser itself does not publish a bottle/formula).

> Two things are NOT the same and are easy to confuse:
> - **`CHANGELOG.md`** (this repo) is hand-written and bundled into the release
>   archives. Editing it does **not** change the GitHub release notes.
> - **GitHub release notes** are auto-generated by GoReleaser from commit
>   subjects. You do not write them by hand.
> Keep `CHANGELOG.md` as the curated human history; let GoReleaser handle the
> auto notes.

## Prerequisites

- On `main`, all intended commits already committed.
- Tests passing (`go test ./...`).
- GitHub CLI authenticated (`gh auth status`).
- `HOMEBREW_TAP_TOKEN` secret configured on the repo (already set; needed by the
  `update-homebrew-tap` job). You do not touch this during a normal release.

A dirty working tree is fine for the **manual** tag commands below as long as
everything you want *in the release* is committed — the tag points at a commit,
so uncommitted local files (e.g. `.claude/settings.local.json`) are not included.
`make tag`/`make release`, however, enforce a clean tree via `check-clean`.

## Versioning

Semantic versioning, bump from the latest tag:
- **Major** (`v1.0.0`): breaking changes
- **Minor** (`v0.47.0`): new features, backward compatible
- **Patch** (`v0.46.1`): bug fixes only

```bash
git describe --tags --abbrev=0        # latest released tag
git log "$(git describe --tags --abbrev=0)"..HEAD --oneline   # what's unreleased
```

## Release Process

### 1. Update `CHANGELOG.md`

Add an entry at the top, under the `## [vX.Y.Z] - YYYY-MM-DD` heading. **Use the
real current date** (`date +%Y-%m-%d`), not a placeholder. Group by area
(`Monitor`, `Features`, `Bug Fixes`, `Documentation`, etc.) matching the style
of existing entries.

```bash
git add CHANGELOG.md
git commit -m "docs: cut vX.Y.Z changelog (short description)"
```

### 2. Verify

```bash
go test ./...
# Optional: dry-run the build without publishing
goreleaser release --snapshot --clean && ls dist/
```

### 3. Push commits, then tag

The tag must be **on top of an already-pushed `main`**. Push commits first, then
the tag — otherwise the released commit only exists under the tag ref and `main`
on GitHub lags behind.

```bash
git push origin main

# Either the Makefile target (clean tree required) ...
make release VERSION=vX.Y.Z      # creates annotated tag AND pushes the tag

# ... or do it by hand (no clean-tree check):
git tag -a vX.Y.Z -m "Release vX.Y.Z: brief description"
git push origin vX.Y.Z
```

> ⚠️ `make release` only pushes the **tag**, not `main`. Always `git push origin
> main` first. (See [`Makefile`](../../Makefile) — the `release` target is
> `tag` + `git push origin "$(VERSION)"`.)

### 4. Verify the release landed

```bash
gh run list --workflow=release.yml --limit 1     # find the run
gh run watch <run-id> --exit-status              # block until done; non-zero on failure
gh release view vX.Y.Z                           # release exists with assets
brew update && brew upgrade td && td version      # Homebrew picked up the bump
```

The Homebrew bump lands a minute or two after the GitHub release (it is the
second job). If `brew upgrade` shows the old version, wait and re-run
`brew update`.

## Agent Runbook

Non-interactive sequence for an automated agent. Avoids blocking/interactive
commands. Replace nothing by hand except the changelog body.

```bash
set -e
cd "$(git rev-parse --show-toplevel)"

# 0. Preconditions
test "$(git rev-parse --abbrev-ref HEAD)" = main
go test ./...

# 1. Decide version (minor bump shown; adjust for patch/major)
LATEST=$(git describe --tags --abbrev=0)         # e.g. v0.46.0
git log "$LATEST"..HEAD --oneline                # review what's shipping
VERSION=vX.Y.Z                                    # set explicitly
DATE=$(date +%Y-%m-%d)

# 2. Edit CHANGELOG.md: add "## [$VERSION] - $DATE" block at the top (use Edit tool).
git add CHANGELOG.md
git commit -m "docs: cut $VERSION changelog"

# 3. Push main BEFORE tagging
git push origin main

# 4. Tag + push tag (manual form tolerates a dirty tree from local-only files)
git tag -a "$VERSION" -m "Release $VERSION"
git push origin "$VERSION"

# 5. Watch CI to completion (non-interactive, fails the step on red)
sleep 5
RUN=$(gh run list --workflow=release.yml --limit 1 --json databaseId -q '.[0].databaseId')
gh run watch "$RUN" --exit-status

# 6. Confirm artifacts
gh release view "$VERSION"
```

Notes for agents:
- Do **not** use bare `gh run watch` (it prompts to pick a run). Pass an explicit
  run id and `--exit-status`.
- The local machine's `td` is upgraded via Homebrew (`brew upgrade td`) or a dev
  build (`make install`). A release does not upgrade the local binary by itself.
- If the `update-homebrew-tap` job fails (e.g. token/permissions), the GitHub
  release still succeeds — re-run just that job from the Actions UI or bump the
  formula manually in `marcus/homebrew-tap`.

## Local Development Builds

```bash
make install                                   # build + install, version from `git describe`
go install -ldflags "-X main.Version=vX.Y.Z" . # explicit version
```

Without ldflags the version reports `dev`/`devel`. Dev versions skip the
update-available check.

## Update Notifications

`td version` checks
`https://api.github.com/repos/marcus/td/releases/latest`, compares `tag_name`
against the running version, and prints an upgrade hint if a newer release
exists. Results are cached for 6 hours in `~/.cache/td/version-cache.json`. Dev
versions (`dev`, `devel+hash`) skip the check.

## Checklist

- [ ] Tests pass (`go test ./...`)
- [ ] `CHANGELOG.md` has a new `## [vX.Y.Z] - <real date>` entry, committed
- [ ] Version follows semver (bumped from `git describe --tags --abbrev=0`)
- [ ] `main` pushed **before** the tag
- [ ] Annotated tag (`-a`) created and pushed
- [ ] `release.yml` run is green (`gh run watch <id> --exit-status`)
- [ ] `gh release view vX.Y.Z` shows binary assets + checksums
- [ ] `marcus/homebrew-tap` `Formula/td.rb` bumped (the `update-homebrew-tap` job)
- [ ] `brew update && brew upgrade td && td version` shows the new version
